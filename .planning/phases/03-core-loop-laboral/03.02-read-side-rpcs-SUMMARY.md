## Decisions Made

- **Projection vs commit split**: `accrueIdleForPibe` is pure projection in get_roster; only `collect_idle` (plan 03.03) writes `last_collected_at` and skill hours. This prevents the T-3-RS-02 double-credit race that would occur if a player calls get_roster immediately before collect_idle.
- **Energy regen persisted by get_roster, not collect_idle**: Energy is a regen-over-time resource (not a soft currency). Persisting it on read keeps the display accurate without needing a separate RPC. It cannot be double-credited because it is clamped at ENERGIA_MAX.
- **pibe.id = pick_id**: `materializePibeFromPick` reuses the pick_id as the canonical pibe.id. This provides a traceability link from the recruit pool pick to the resulting pibe in storage (supports LAB-RECRUIT-RACE invariant in plan 05).
- **Rank transition does NOT trigger immediate Mesa recompute**: `checkRankTransition` sets `mesa_recompute_pending = true` and returns. Actual recompute happens lazily on the next `get_barra_state` call that satisfies the 5-min debounce, or on the hourly `bb_mesa_recompute_1h` cron. This prevents the Mesa thrash described in T-3-RS-06.
- **get_barra_state accepts optional club_id**: Payload `{club_id: "..."}`  allows viewing other clubs' barra state — prepared for Phase 5 cross-club viewing. Defaults to caller's club from server profile (no client trust for club_id — T-3-RS-09).
